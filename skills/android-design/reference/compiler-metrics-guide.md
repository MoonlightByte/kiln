# Compose Compiler Metrics Guide

How to generate, read, and interpret Jetpack Compose compiler stability reports.

---

## Overview

Compiler metrics are build-time reports generated by the Compose Compiler (v1.2.0+) that reveal how the Compose Compiler optimizes your composables. They show which functions are **restartable** vs **skippable**, helping you understand recomposition performance.

**Goal**: Maximize skippable composables while maintaining proper recomposition boundaries.

---

## Key Concepts

### Restartable Functions

**Definition**: Composables that Compose can re-invoke when inputs change.

```kotlin
@Composable
fun Counter(count: Int) {  // Restartable: can restart when count changes
    Text("Count: $count")
}
```

**What it means**:
- Compose marks this as a "scope" where state changes trigger recomposition
- The function body will re-execute if any input parameter changes
- Nearly all `@Composable` functions are restartable—this is foundational to Compose

**In metrics**: Listed in `_restartable.csv`

---

### Skippable Functions

**Definition**: Composables that Compose can bypass entirely if their parameters haven't changed.

```kotlin
@Stable  // <- Stability annotation
data class EventCard(val id: String, val title: String)

@Composable
fun EventCardRow(event: EventCard) {  // Skippable: if 'event' object identity unchanged, skip
    Card {
        Text(event.title)
    }
}
```

**What it means**:
- Compose checks if parameters have changed before re-executing
- If all parameters are identical (by reference or immutable structure), the function is skipped entirely
- Child composables are also skipped (no cascading recomposition)
- Critical for performance in frequently-recomposed trees

**In metrics**: Listed in `_composables.csv` with `skippable=true`

**Impact**: A skipped parent = all children skipped (no recomposition cascade)

---

### Stability

**Definition**: Whether Compose can determine if a parameter type changes between recompositions.

#### Stable Types (Skippable)
- Primitives: `Int`, `String`, `Boolean`, `Float`, etc.
- Immutable data classes annotated `@Immutable`
- Collections from `kotlinx.collections.immutable` (`ImmutableList`, `ImmutableSet`)
- `StateFlow<T>`, `Flow<T>`
- Composable lambdas (functions)

#### Unstable Types (NOT Skippable)
- Mutable collections: `List<T>`, `Set<T>`, `Map<K, V>` (Compose can't prove immutability)
- Data classes without `@Immutable` annotation
- Classes with mutable properties
- Custom types without stability annotations

## Generating Metrics

### Enable Compiler Metrics in Gradle

```gradle
// build.gradle.kts (module level)

android {
    // Enable metrics generation
    tasks.withType<KotlinCompile>().configureEach {
        kotlinOptions {
            // For local metrics (saved to build/compose_metrics)
            freeCompilerArgs += listOf(
                "-P", "plugin:androidx.compose.compiler.plugins.kotlin:metricsDestination=${project.buildDir}/compose_metrics"
            )
            // For JSON reports
            freeCompilerArgs += listOf(
                "-P", "plugin:androidx.compose.compiler.plugins.kotlin:reportsDestination=${project.buildDir}/compose_reports"
            )
        }
    }
}

// Gradle project-level settings (optional)
android {
    composeOptions {
        kotlinCompilerExtensionVersion = "1.5.3"  // Match Compose version
    }
}
```

### Build and Generate

```bash
# Clean and rebuild to regenerate metrics
./gradlew clean assembleDebug

# Metrics appear in:
# build/compose_metrics/
# ├── app_composables.csv
# ├── app_composables.txt
# ├── app_classes.txt
# ├── app_composables_restartable.csv
# └── app_composables_restartable.txt
```

## Reading the Reports

### File Breakdown

#### 1. `*_composables.csv` (Most Important)

Header row:
```
package,name,composable,restartable,skippable,inline
```

Example data:
```
com.example.app.ui,EventListScreen,EventListScreen,true,false,"{"
com.example.app.ui,EventCard,EventCard,true,true,"{"
com.example.app.ui,EventTitle,EventTitle,true,true,"{"
com.example.app.components,Button,Button,true,true,"{"
```

**Columns**:
- `package`: Module/package of composable
- `name`: Function name
- `composable`: Always true (these are all composables)
- `restartable`: Can Compose restart it? (almost always true)
- `skippable`: Can Compose skip it if params unchanged? (want this TRUE)
- `inline`: Is the function inlined? (optimization detail)

**Analysis Goal**: Maximize `skippable=true` entries.

#### 2. `*_composables.txt` (Details)

```
com/example/app/ui/EventListScreen.kt
  "EventListScreen" (composable=true, restartable=true, skippable=false)
    STABLE: String
    UNSTABLE: List<Event>  <- This makes EventListScreen unskippable!

  "EventCard" (composable=true, restartable=true, skippable=true)
    STABLE: Event  <- Marked @Immutable
    STABLE: () -> Unit
```

**Why EventListScreen is unskippable**:
- Parameter `events: List<Event>` is unstable (mutable List)
- Even if contents don't change, Compose recomposes because it can't prove stability

**Solution**:
```kotlin
// BEFORE - Unskippable
@Composable
fun EventListScreen(events: List<Event>) {
    LazyColumn { /* ... */ }
}

// AFTER - Skippable
@Composable
fun EventListScreen(events: ImmutableList<Event>) {
    LazyColumn { /* ... */ }
}
```

#### 3. `*_classes.txt` (Type Stability)

```
com/example/app/data/Event
  UNSTABLE: private val attendees: List<User>  <- Mutable List
  UNSTABLE: private val tags: Set<String>      <- Mutable Set
  STABLE: val id: String
  STABLE: val title: String

com/example/app/data/EventList
  STABLE: class (marked @Immutable)
  STABLE: val items: ImmutableList<Event>  <- Using immutable collections
```

**Key Issue**: Even one mutable field makes the entire class unstable.

**Fix**:
```kotlin
// BEFORE - Unstable (NEVER marked @Immutable)
data class Event(
    val id: String,
    val title: String,
    val attendees: List<User>  // <- Mutable
)

// AFTER - Stable
@Immutable
data class Event(
    val id: String,
    val title: String,
    val attendees: ImmutableList<User>  // <- Immutable, can mark @Immutable
)
```

## Optimization Workflow

### Step 1: Identify Unskippable Composables

```bash
# Extract all unskippable composables
grep "skippable=false" build/compose_metrics/*_composables.csv > /tmp/unskippable.txt

# Count them
wc -l /tmp/unskippable.txt
```

### Step 2: Analyze Why They're Unskippable

```bash
# Find the composable in the .txt file
grep -A 5 "EventListScreen" build/compose_metrics/*_composables.txt

# Output:
# "EventListScreen" (composable=true, restartable=true, skippable=false)
#   UNSTABLE: List<Event>
#   UNSTABLE: (Event) -> Unit
```

### Step 3: Make Them Skippable

#### Option A: Use Immutable Collections

```kotlin
import kotlinx.collections.immutable.ImmutableList
import kotlinx.collections.immutable.persistentListOf

@Composable
fun EventList(
    events: ImmutableList<Event>,  // <- Changed to ImmutableList
    onSelectEvent: (Event) -> Unit
) {
    LazyColumn {
        items(events, key = { it.id }) { event ->
            EventCard(event, onSelectEvent)
        }
    }
}
```

#### Option B: Mark Data Class @Immutable

```kotlin
import androidx.compose.runtime.Immutable

@Immutable  // <- Tell Compose this is stable
data class Event(
    val id: String,
    val title: String,
    val attendees: ImmutableList<User>  // <- Must use immutable collections
)
```

**Requirement**: All fields must be provably stable (primitives, other @Immutable classes, immutable collections).

#### Option C: Use Stable Interface

```kotlin
interface EventProvider {
    fun getEvents(): ImmutableList<Event>
}

@Composable
fun EventList(provider: EventProvider) {
    val events = remember(provider) {
        provider.getEvents()
    }
    // Now skippable if provider identity doesn't change
}
```

### Step 4: Rebuild and Verify

```bash
./gradlew assembleDebug

# Re-check metrics
grep "EventList.*skippable=true" build/compose_metrics/*_composables.csv
# Should now show skippable=true
```

## Case Study: Optimizing a Feed Screen

### Before Optimization

```kotlin
// data/Event.kt
data class Event(  // <- UNSTABLE: no @Immutable, uses List<>
    val id: String,
    val title: String,
    val tags: List<String>,
    val attendees: List<User>
)

// ui/FeedScreen.kt
@Composable
fun FeedScreen(
    events: List<Event>,  // <- UNSTABLE: mutable List
    onSelectEvent: (Event) -> Unit  // <- Stable: lambda
) {
    LazyColumn {
        items(events) { event ->
            EventCard(event, onSelectEvent)  // <- FeedScreen unskippable
        }
    }
}

@Composable
fun EventCard(event: Event, onSelect: (Event) -> Unit) {
    Card(
        onClick = { onSelect(event) },
        modifier = Modifier.fillMaxWidth()
    ) {
        Text(event.title)
        Text(event.tags.joinToString(", "))  // <- Recomposed every time
    }
}
```

**Metrics Report**:
```
FeedScreen       -> restartable=true, skippable=false  (UNSTABLE: List<Event>)
EventCard        -> restartable=true, skippable=false  (UNSTABLE: Event)
```

**Performance Impact**: Every parent recomposition cascades to all event cards (n items recomposed).

### After Optimization

```kotlin
// data/Event.kt
import androidx.compose.runtime.Immutable
import kotlinx.collections.immutable.ImmutableList
import kotlinx.collections.immutable.persistentListOf

@Immutable  // <- Now STABLE
data class Event(
    val id: String,
    val title: String,
    val tags: ImmutableList<String> = persistentListOf(),  // <- Use immutable
    val attendees: ImmutableList<User> = persistentListOf()
)

// ui/FeedScreen.kt
import kotlinx.collections.immutable.ImmutableList

@Composable
fun FeedScreen(
    events: ImmutableList<Event>,  // <- Changed to ImmutableList
    onSelectEvent: (Event) -> Unit
) {
    LazyColumn {
        items(events, key = { it.id }) { event ->
            EventCard(event, onSelectEvent)
        }
    }
}

@Composable
fun EventCard(event: Event, onSelect: (Event) -> Unit) {
    Card(
        onClick = { onSelect(event) },
        modifier = Modifier.fillMaxWidth()
    ) {
        Text(event.title)
        Text(event.tags.joinToString(", "))
    }
}
```

**New Metrics Report**:
```
FeedScreen       -> restartable=true, skippable=true   (STABLE)
EventCard        -> restartable=true, skippable=true   (STABLE)
```

**Performance Impact**:
- Parent recomposition: No change (still recomposes)
- Child items: LazyColumn + stable Event objects = only affected items recompose
- Smooth scrolling preserved

## Practical Metrics Interpretation

### Scenario 1: Unskippable Dialog

```
# From metrics
DialogContent   -> skippable=false
```

**Cause**:
```kotlin
@Composable
fun DialogContent(data: DialogData) {  // DialogData is unstable
    Text(data.title)
}
```

**Fix**:
```kotlin
@Immutable
data class DialogData(val title: String, val message: String)

@Composable
fun DialogContent(data: DialogData) {  // Now stable -> skippable
    Text(data.title)
}
```

---

### Scenario 2: List Items Recomposing on Scroll

```
# Scroll causes parent recomposition
UserListScreen -> skippable=false (UNSTABLE: List<User>)
UserCard       -> skippable=false (UNSTABLE: User)
```

**Cause**: Both unstable, so all items recompose on any parent change.

**Fix**:
```kotlin
@Immutable
data class User(val id: String, val name: String)

@Composable
fun UserListScreen(
    users: ImmutableList<User>  // Immutable list
) {
    LazyColumn {
        items(users, key = { it.id }) { user ->
            UserCard(user)  // Now skippable
        }
    }
}
```

---

### Scenario 3: Lambda Callbacks (Always Stable)

```
# Lambdas are stable by default
SearchScreen -> skippable=true
    STABLE: String
    STABLE: (String) -> Unit  <- Lambda callbacks are stable!
```

**Good News**: Callback lambdas don't prevent skippability. Safe to pass.

## Common Issues & Solutions

| Issue | Cause | Solution |
|-------|-------|----------|
| Composable unskippable | Uses `List<T>` parameter | Change to `ImmutableList<T>` |
| Data class unstable | Missing `@Immutable` | Add `@Immutable`, use immutable collections inside |
| LazyColumn items recompose | Item class unstable | Mark with `@Immutable`, use stable key |
| All children recompose | Parent unskippable | Make parent stable (fix collections/annotations) |
| Performance regression | Too many unskippable at high level | Add `remember()` for expensive subtrees |

## Best Practices

### 1. Annotate Data Classes

```kotlin
@Immutable
data class MyModel(
    val id: String,
    val data: ImmutableList<String>
)
```

### 2. Use Immutable Collections

```kotlin
// Dependency
dependencies {
    implementation("org.jetbrains.kotlinx:kotlinx-collections-immutable:0.3.6")
}

// Usage
val list: ImmutableList<Item> = persistentListOf(item1, item2)
val set: ImmutableSet<String> = persistentSetOf("a", "b")
```

### 3. Remember Expensive Composables

```kotlin
// If you can't make a composable skippable, cache its output
val cachedContent = remember(key) {
    ExpensiveComposable()  // Only called when key changes
}
```

### 4. Review Metrics Regularly

```bash
# Add to CI/CD
./gradlew clean assembleDebug
find build/compose_metrics -name "*_composables.csv" -exec grep "skippable=false" {} \;
```

### 5. Test with MacroBenchmarks

```kotlin
// Verify skippability improves real-world performance
@Test
fun scrollFeed_smooth() = benchmarkRule.measureRepeated(
    metrics = listOf(FrameTimingMetric()),
) {
    // Scroll and measure frame times
}
```
